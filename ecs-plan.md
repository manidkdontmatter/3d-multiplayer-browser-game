# “Anything Can Be Anything” ECS system that supports complete flexibility for players to use various UI menus while in game to create new objects that might be anything

This game is intended to support an “anything can be anything” simulation architecture similar in spirit to games like *Caves of Qud*, *Dwarf Fortress*, *Cataclysm: Dark Days Ahead*, and other complex systemic games. It needs to have the most flexibility possible.

An example of what this system will be used for in gameplay: Players will be able to create characters, items, abilities, and more, while in game, by using various UI like a character creator ui, item creator ui, and ability creator ui, etc, but under the hood there is no real concept of characters, items, or abilities, it is far more flexible than that, the different UIs that allow the player to create things are only separated into those categories on a superficial UI level, a presentation layer, to make creating things more guided and comprehensible for players, but under the hood it is completely flexible and not bound to such concepts as characters, items, or abilities, etc. The UIs to create different types of things not only exist as a purely presentation layer but will also limit what components and other customizations are available to the player within the menu, meaning for creating an item it will only display components and such that the game has at a superficial presentation level deemed usable for creating items using that UI, but the underlying system is not bound to types as such. Some examples of what players will create in this game using this underlying system is: items, abilities, characters, pets, npcs, bioengineered mutants, robots, companions, and anything else I feel like adding. But this document is not about implementing the UIs that allow the player to create things, that is a separate thing to be done later, this is about creating the underlying systems that provide the flexibility to do such things in the first place. At the end of the day everything is an entity and anything can be anything.

however, i will tell you that it is intended for what the player creates using those UIs will be data drive design data, this data persists. for example if the player uses the ability creator ui to create an ability after they customize it how they want it it will create a data file that describes the ability, the purpose of this is that not only the player who created the ability can use the ability, they can teach the ability to other players, but ultimately all players with that ability are referencing the same data file (blueprint?). the ecs representation of the ability so it can actually be used by players during runtime is created by interpreting the data file (blueprint?). the various creator ui's the player uses to create things must validate what they intend to create before they finally create it. the blueprint must be validated before being created, so that it follows certain rules for its type (character, item, ability, etc, remember these are just presentation layer types). although the ui to create things is client side, when they try to create it it is validated on the server, and if it passes the server creates it.

almost all creator ui's share a few things in common, they let you create an overall type (examples: character, item, ability, and more), then you choose a subtype (for item, there would be weapon, clothing, etc), each subtype has subtype specific stats and attributes you can change to customize it, every subtype may have its own completely unique stat names and attribute names. stats are numbers you can put points into. attributes are upsides and downsides you can pick to add to it. this is all a presentation layer in the ui, under the hood it is all ecs i think, and those stats/attributes (components?) are not limited to any concept of a type or subtype under the hood, to keep everything completely flexible. but although it is a presentation layer, the server still validates what is about to be created so that it actually follows the defined rules of that presentation layer before the server will allow it to be created.
